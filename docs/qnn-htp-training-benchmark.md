# QAIRT 2.48 HTP-assisted linear-training benchmark

## Scope

This benchmark measures reproducibility and performance of PhoneLM linear regression on
nubia Z80 Ultra / NX741J (SM8850, HTP V81, Android 16). The formal SDK is
QAIRT 2.48.40.260702, Build ID `2.48.40.260702151143`, QNN Core API 2.37.0,
HTP Backend API 5.48.0, Android NDK 26.2.11394342, and CMake 3.22.1.
QAIRT 2.47 artifacts are forbidden by the APK audit.

The execution split is exact:

- HTP: fixed-shape float32 MatMul forward.
- CPU: loss, gradient, SGD optimizer, and copying the updated weight into the
  APP_WRITE client buffer.
- QNN: the same finalized graph is executed again after the APP_WRITE buffer update.

Backward and optimizer are not executed on HTP. CPU fallback is forbidden and is
never accepted as success.

## Dataset and reproducibility

The multi-batch problem is `Y_i = X_i W*`, without bias. The dataset, true weight,
initial weight, and shuffle stream are generated by `std::mt19937` from the requested
seed. The first `input_dim` rows of `X` are a 0.25-scaled identity matrix when
`sample_count >= input_dim`; therefore the vertically concatenated dataset has full
column rank by construction. Remaining rows are uniform in [-0.25, 0.25].

CPU and HTP runs independently regenerate identical data and per-epoch shuffle order.
A single fixed batch with `batch_size < input_dim` has rank at most `batch_size`;
many weights produce the same batch output, so loss can fall without uniquely
recovering `W*`. Multi-batch training removes that underdetermination.

## Modes

The legacy modes remain unchanged:

- `QNN_CPU_LINEAR_TRAINING`
- `QNN_HTP_LINEAR_TRAINING`
- `QNN_LINEAR_GRADIENT_CHECK`

New correctness modes are `QNN_CPU_MULTIBATCH_TRAINING` and
`QNN_HTP_MULTIBATCH_TRAINING`. They compare every forward with the CPU reference
and record epoch loss and weight error.

New performance modes are `QNN_CPU_TRAINING_BENCHMARK` and
`QNN_HTP_TRAINING_BENCHMARK`. They omit per-step reference MatMul, string
generation, file writes, and debug logging. They retain loss, gradient, optimizer,
APP_WRITE update, finite checks, periodic reference checks, and final verification.
Reference checks are timed separately and excluded from `full_step`.

## Timing boundaries

C++ uses `std::chrono::steady_clock`. Every reported distribution contains min,
median, mean, population standard deviation, p90, p95, and max. Warmup samples and
the first HTP execute are excluded from steady-state benchmark distributions.

`full_step` covers mini-batch preparation/copy, forward, loss, gradient, optimizer,
weight-buffer copy, and APP_WRITE client-buffer rebinding. Reference validation,
epoch-wide evaluation, record insertion, CSV formatting, and file I/O are outside
`full_step` and separately recorded or reflected only in total run time.

QNN tensors use `QNN_TENSORMEMTYPE_RAW` app-owned buffers. No explicit pre/post
sync API is called. `QnnGraph_execute` is synchronous in this path: output is read
only after it returns, so its call time includes backend transfer and completion wait.
`output_copy` is the host copy from the app-owned QNN output vector to the CPU loss
matrix plus output-buffer binding. `weight_buffer_copy` is the float copy into
runtime-owned APP_WRITE storage; `runtime_weight_update` is the distinct client
buffer rebinding/bookkeeping step. Input/output buffer allocation is reused after the
first step.

The former unexplained full-step gap came mainly from two correctness calculations
inside each step: a CPU reference MatMul and a complete CPU
`forwardBackward + SGD` reference path. They are no longer included in benchmark
`full_step`.

Initialization is reported separately as backend, device, context, graph create,
graph finalize, first execute, and aggregate initialization time. The current runtime
creates and destroys these objects once per app run; it does not claim a cross-run
warm context.

## Crossover definitions

The report distinguishes:

- forward-only crossover: CPU MatMul median versus HTP graph-execute median;
- full-step crossover: median algorithmic training step;
- total-run crossover: total time including initialization and final verification.

When `CPU_step > HTP_step`, the estimated recovery point is:

`ceil(HTP_initialization / (CPU_step - HTP_step))`.

If the denominator is not positive, the report leaves break-even steps empty
(no break-even under that condition).

The CPU MatMul is the existing scalar row-major C++ implementation, compiled as the
debug APK native configuration shown by the build log. It has no OpenMP integration
in this path and must not be described as the device CPU's optimized upper bound.

## Thermal and run-order policy

The runner records battery temperature, Android thermal status, battery level,
charging status, and elapsed realtime before and after each run. It alternates CPU/HTP
order between repetitions. Android thermal status is authoritative: status SEVERE
or higher pauses new work for cooling. The runner does not root the device, change
SELinux, fix frequencies, disable throttling, write sysfs, or alter persistent device
settings.

## Reproduction

```powershell
.\scripts\run_qnn_training_benchmarks.ps1 `
  -QairtSdkRoot 'C:\Qualcomm\AIStack\QAIRT\2.48.40.260702' `
  -ExpectedBuildId '2.48.40.260702151143' `
  -Repetitions 3 `
  -Seeds 20260710,20260711,20260712,20260713,20260714 `
  -Phase Seeds
```

Run `-Phase Stage1`, inspect `break-even.csv`, then run `-Phase Refine`.
Use `-Phase Extended` only when temperature and estimated runtime are reasonable.
Use the same `-ReportRoot` to resume; successful JSON runs are skipped unless
`-RerunSuccessful` is supplied.

Generated APKs, reports, CSV, JSON, and logs remain under `build/` and must not be
committed. APK audit is required before device execution and verifies the V81 Stub,
unsigned V81 Skel, SDK hashes, and absence of known QAIRT 2.47 identifiers.
